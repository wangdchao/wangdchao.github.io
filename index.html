nginx安装

确保上一步能够成功连接到远程服务器的基础上。在Xshell中输入以下指令让远程服务器执行。

sudo apt-get update

sudo apt-get install nginx

以上无误的话，nginx便安装好了，你可以输入nginx -v来查看所安装的nginx版本。如果相似nginx命令不识别，则表明nginx安装失败。

vue组件与路由

目前对于web网页前端开发来说，具有很多的前端框架，其中vue所提供的的前端框架在最近几年算是比较热门的了。这里，我就想总结一下我自己对于一下在vue框架在搭建单页网页应用时组件与路由，当然，你也可以到一些具体的教程网站上去学习，比如vue的官网、菜鸟教程中都有vue的使用教程。

至于vue的安装以及初始项目的创建，可以参考这里。

基于vue的单页和多页

在进行web网页的书写时，我们都会涉及到多个页面之间的切换等，这时，可以采用多页网页应用，虽然vue本身基于vue-cli是基于单页的，但是也可以更改成多页，查看vue构建多页应用可以参考这个链接。

当然，采用vue的组件和路由也可以实现单个页面的多个视图之间的切换，只不过是将原来的多页面更换成vue中的组件，在利用路由进行切换即可。

vue的路由用来实现各个子视图之间的跳转。而vue的路由实现便是通过vue-router这个库来实现的。当你需要使用vue-router是，你需要采用npm指令来安装这个库，并在你的vue项目中引用：

import Router from ‘vue-router’

在指定的管理路由与组件的文件(通常是在src文件下新建router文件夹，并新建index.js,依次来管理路由)使用如下的：

接着需要了解到vue里面的组件。在vue项目里，一个页面几乎是由组件及其相互关系来构建的，vue项目就好像是一栋房子，组件就是这房子的砖块，由‘砖块’构成总的项目，其中的结构如下图：

import Recharge from ‘../components/User/Recharge.vue’

import Withdraw from ‘../components/User/Withdraw.vue’

路由与组件的结合

视图切换就是由某个组件跳到某个组件，这个路由跳转有两种形式。

  • 直接在html利用\<router-link>元素来跳转，其中定义了跳转的路径，就是在上面的index.js中定义的与组件对应的路径名。当点击这个链接是，就会跳转到指定的组件，而跳转到的组件显示的位置，由\组件决定。所以这两个元素是搭配使用的。

  • 在组件对应的js中，利用函数中的router实例调用push(‘路由路径’),来实现跳转。此时可以不再需要\<router-link>标签。

    this.$router.push(‘/User/Part/myTodoTask’)

组件之间的传值

组件之间可能涉及到传值，如父组件需要将本地一个属性的值传到子组件，子组件有时也需要传个值给父组件以期望父组件进行相应的更新。当然，对于路由跳转是，也许新跳转的组件需要一些初始的值来初始化自己数据的定义或初始的网络请求的参数。针对这两种情况各有各的方法。

对于路由跳转到一个新的组件页面时通过路由传参

路由传参有两种方式：

  • 在调用$router的push函数时，除了path外，新加一个params字段，该字段是一个json类型的，也可以是一个单值。这个params的值就会通过路由传到新的路由跳转到的组件中。如下：

    此时在新的组件主需要调用$route.params便可以获得上面定义的params的值。

  • 除了params之外，还可以通过query这个关键值传值，而它与params的区别在params的值会在浏览器的路径名中显示出来。而query则不会。

    同样调用\$route.query便可以得到query中的json值。

父子组件之间的传值

优势两个组件的关系是固定的，并不会采用路由来跳转，所以这时在父子之间的传值，如下。

  • 父组件向子组件传值：这种情况需要在子组件中用到props.如下图，据题参考：props。

  • 子组件向父组件传值：这种利用了事件的冒泡原理，有子元素发生的事件会一层层上浮，所以在子组件自定义一个事件，主动触发这个事件，父组件接受到该事件后进行相应的处理，从而实现自子组件向父组件的传值。主动触发事件，调用函数\$emit()，即可，\$emit(‘enlarge-text’, 0.1)，第一个参数便是自定义事件名，第二个是事件携带的参数。在父组件中只需要注册号该事件的监听器就行了，并配置相应的相应函数即可。可以参考如下：参考。

其它传值

其实对于很多组件页面都需要用到的值，可以利用浏览器提供的缓存来进行，而不要进行复杂的路由传值、父子传值。

这个缓存推荐使用SessionStorage.这个缓存只要是在浏览器同一个窗口的页面都可以访问。而具体使用起来很简单。四个操作：

  • sessionStorage.setItem(“data”, msg.value) ：将一个值存到缓存中，访问名称为’data‘。插入一个项目

  • sessionStorage.getItem(“data”)：获取访问名称为’data‘的值

  • sessionStorage.removeChild(msg); ：删除访问名称为msg的值的这条项目

  • sessionStorage.clear()：清空缓存中的数据。

用例建模 - 绘制用例图

简答题

1.用例的概念

用例功能要求或行为要求，以指示系统将做什么，是相关成功和失败场景的集合，描述了使用系统来达到某个目标的参与者。

2.用例和场景的关系？什么是主场景或 happy path？

用例表示场景的集合：主方案，加上零或多个备选方案。

主场景对应于主要的系统交互，通常是“成功”场景。

3.用例有哪些形式？

  • Brief：简洁的一段总结，通常是主要的成功情景.
  • Casual：非正式段落格式。涵盖各种场景的多个段落。
  • Fully：把所有的步骤和变体都写得很详细，还有一些支持的部分，比如先决条件和成功保证。

4.对于复杂业务，为什么编制完整用例非常难？

复杂业务，本身的场景逻辑比价复杂，场景较多且各个场景交叉关联。由此增加了对于场景的归类和充分提取的难度。如果要想编制完整的用例，需要充分了解到业务逻辑，对业务本身的结构有清晰的认识。这对于编写完整用例所用的时间又提出了很高的要求。

5.什么是用例图？

用例图，用于显示用例和参与者的名称以及它们之间的关系。这给出了一个很好的系统及其环境的上下文图。

6.用例图的基本符号与元素？

基本的元素包括：参与者（actor）、系统（System）和子系统、用例（User case）、信息流（一条黑色实线）、注释（note）、关系、项目等。

关系分为关联、泛化、包含、拓展几种，分别由下图中带符号的实线或虚线定义。

下图是对应元素的符号图：

7.用例图的画法与步骤

画法与步骤：

  • 明确系统功能与目标，确定系统同边界与名称（画带系统名称矩形方框，标定系统边界）

  • 明确系统参与者，如用户、时钟等（画出对应的参与者的符号放在系统框的左侧）

    • 参与者识别：系统使用人、支持系统运行的系统或组织。

  • 确认用例：通常结合系统功能，明确参与者与系统可能的交互场景，利用通过事件驱动的方法识别用户案例。关于用户案例识别：

    • 以主要参与者目标驱动
    • 收集主要参与者的业务事件
    • 必须满足以下准则：Boss test、EBP test、Size test

    对于不同的用例，画出对应的用例符号放在系统框内。

  • 明确用例之间的关系，用例与子用例的关系，在拥有相应关系的用例符号之间画上相应的关系符号，如包含关系、拓展关系符号等。

  • 确认参与者与用例之间的关联，通常为参与者与用例之间用实线相连。

  • 明确外部关联的系统，也用实线相连。

8.用例图给利益相关人与开发者的价值有哪些？

帮助开发和测试团队以一种可视化的方式理解系统的功能需求。每个椭圆代表一个用例，这样方便进行需求分析，不会遗漏，在进行测试时，可以根据用例生成测试用例。明确系统的业务范围、服务对象（角色）、外部系统与设备；帮助识别技术风险，提前实施关键技术原型公关与学习；易于评估项目工作量，合理规划迭代周期，规划人力需要。

简答题

简述瀑布模型、增量模型、螺旋模型（含原型方法）的优缺点（从项目特点、风险特征、人力资源利用角度思考）

瀑布模型：

优点：

  • 降低软件开发的复杂程度，提高软件开发过程的透明性，提高 软件开发过程的可管理性。
  • 推迟软件实现，强调在软件实现前必须进行分析和设计工作。
  • 以项目的阶段评审和文档控制为手段有效地对整个开发过程进 行指导，保证了阶段之间的正确衔接，能够及时发现并纠正开 发过程中存在的缺陷，使产品达到预期的质量要求。

缺点：

  • 强调过程活动的线性顺序。
  • 缺乏灵活性，尤其无法解决软件需求不明确或不准确的问题。
  • 风险控制能力较弱。
  • 瀑布模型中的软件活动是文档驱动的，当阶段之间规定过多的 文档时，会极大地增加系统的工作量。
  • 管理人员如果仅仅以文档的完成情况来评估项目完成进度，往 往会产生错误的结论。

增量模型：

优点：

  • 增强客户对系统的信心
  • 降低系统失败风险
  • 提高系统可靠性
  • 提高系统的稳定性和可维护性

缺点：

  • 建立初始模型时，作为增量基础的基本业务服务的确定有一定 难度
  • 增量粒度难以选择

螺旋模型：

优点

  • 设计上的灵活性,可以在项目的各个阶段进行变更
  • 以小的分段来构建大型系统,使成本计算变得简单容易
  • 客户始终参与每个阶段的开发,保证了项目不偏离正确方向以及项目的可控性
  • 随着项目推进,客户始终掌握项目的最新信息 , 从而他或她能够和管理层有效地交互
  • 客户认可这种公司内部的开发方式带来的良好的沟通和高质量的产品

缺点：

  • 很难让用户确信这种演化方法的结果是可以控制的。建设周期长，而软件技术发展比较快，所以经常出现软件开发完毕后，和当前的技术水平有了较大的差距，无法满足当前用户需求

简述统一过程三大特点，与面向对象的方法有什么关系？

三大特点：

  • 用户为中心（描述用户使用产品的故事）
  • 架构为重心（软件产品分解的领域最佳实践）
  • 风险驱动（早期暴露风险）

统一过程是一个面向对象且基于网络的程序开发方法论。

简述统一过程四个阶段的划分准则是什么？每个阶段关键的里程碑是什么？

四个阶段：其划分的依据是各个阶段实行的工作内容，初始阶段：确定大体构想、确定系统范围，建立目标里程碑；细化阶段：体系结构设计，完成系统开发计划，建立结构里程碑；构建阶段：对遗留下的风险较低和比较简单的元素进行迭代实现，准备部署。构造产品，并继续演进需求、体系结构和计划，直到产品完成，建立运行能力里程碑；移交阶段：系统部署与测试，移交用户,建立发布里程碑。

软件企业为什么能按固定节奏生产、固定周期发布软件产品？它给企业项目管理带来哪些好处？

软件企业大都采用迭代增量的软件开发形式，开发过程分为多个阶段，每个阶段有其目标与里程碑。在固定的时间划分内，实现固定的项目目标。完成一个里程碑，进入下一个阶段。因而可以有序的进行项目开发。

因此企业在企业在使用UP时依据各个迭代过程可以有固定的节奏生产、固定时间发布软件产品。

系统分析第一次作业

简答题

软件工程的定义

软件工程是（1）将系统化的、规范的、可度量的方法应用于软件的开发、运行和维护，即将工程化方法应用于软件；（2）在（1）中所述方法的研究。

解释导致 software crisis 本质原因、表现，述说克服软件危机的方法

软件危机其原因，衔接到硬件的整体复杂度，与软件开发流程。计算机的发展导致了软件危机。由于软件本省对于硬件的要求和自身开发及运维的复杂性和高成本性是其本质原因，具体原因体现如下：

  • 软件的规模越来越大，结构越来越复杂
  • 软件开发管理困难而复杂
  • 软件开发费用不断增加
  • 软件开发技术落后
  • 生产方式落后
  • 开发工具落后，生产率提高缓慢

软件危机具体表现有：

  • 项目运行超出预算
  • 项目运行超过时间
  • 软件质量低落
  • 软件通常不匹配需求
  • 项目无法管理，且代码难以维护

克服危机需要我们做到要克服软件危机，就要认真分析软件危机的原因，探索用工程的方法进行软件生产的可能性，即用现代工程的概念、原理、技术和方法进行计算机软件的开发、管理、维护和更新。

软件生命周期

软件生命周期是指软件的产生直到成熟的全部过程。生命周期是事物发展的客观规律，软件同样存在生命周期。早期的软件生命周期往往是说“软件从计划、需求开始，经历分析设计、实现、部署、维护，直到最后逐渐消亡的”。这是受到了第一个软件生命周期模型—瀑布模型影响，上述语句实质上简要的描述了瀑布型生命周期。 现在的软件生命周期不再只考虑瀑布型生命周期，另外常见的软件生命周期模型有原型模型、螺旋模型、迭代模型。所以现在的软件生命周期说明应当不再包括瀑布型生命周期中的典型阶段。
因此，现在对软件生命周期及软件生命周期模型采用如下定义：

  • 软件生命周期是指软件的产生直到成熟的全部过程
  • 软件生命周期模型是指人们为开发更好的软件而归纳总结的软件生命周期的典型实践参考

最近几年来，给软件生命周期带来最多活力的是敏捷软件开发，使得这个领域呈现出勃勃生机，出现了一些更好响应变化、迎接竞争的生命周期模型。

SWEBoK 的 15 个知识域（An Overview of the SWEBOK Guide 请中文翻译其名称与简短说明）。

  • Software Requirements（软件要求）：软件要求关注软件需求的启发，协商，分析，规范和验证。
  • Software Design（软件设计）：软件设计过程是软件工程生命周期活动，其中分析软件需求以产生软件内部结构及其行为的描述，其将作为其构造的基础。
  • Software Construction（软件构建）：软件构建是指通过结合详细设计，编码，单元测试，集成测试，调试和验证来详细创建工作软件。
  • Software Testing（软件测试）：软件测试涉及在有限的测试用例集上针对预期行为动态验证程序的行为。
  • Software Maintenance（软件维护）：软件维护包括增强现有功能，调整软件以在新的和修改的操作环境中运行，以及纠正缺陷。
  • Software Configuration Management（软件配置管理）：软件配置管理（SCM）是在不同时间点识别系统配置的规则，用于系统地控制配置的改变，以及在整个软件生命周期中维持配置的完整性和可追溯性。
  • Software Engineering Management（软件工程管理）：软件工程管理涉及规划，协调，测量，报告和控制项目或程序，以确保软件的开发和维护是系统化的，规范化的和量化的。
  • Software Engineering Process（软件工程过程）：软件工程KA关注软件生命周期过程的定义，实施，评估，测量，管理和改进。涵盖的主题包括流程实施和变更（流程基础架构，流程实施和变更模型以及软件流程管理）; 流程定义（软件生命周期模型和流程，流程定义，流程适应和流程自动化的符号）; 过程评估模型和方法; 测量（过程测量，产品测量，测量技术和测量结果的质量）; 和软件处理工具。
  • Software Engineering Models and Methods（软件工程模型和方法）：软件工程模型和方法解决了涵盖多个生命周期阶段的方法; 其他知识域涵盖特定生命周期阶段的特定方法。涵盖的主题包括建模（软件工程模型的原理和属性;语法与语义与不变量;前置条件，后置条件和不变量）; 模型类型（信息，结构和行为模型）; 分析（分析正确性，完整性，一致性，质量和相互作用;可追溯性;以及权衡分析）; 和软件开发方法（启发式方法，形式方法，原型方法和敏捷方法）。
  • Software Quality（软件质量）：软件质量是许多SWEBOK V3 知识域中普遍存在的软件生命周期问题。此外，软件质量KA还包括软件质量的基础知识（软件工程文化，软件质量特性，软件质量的价值和成本以及软件质量改进）; 软件质量管理流程（软件质量保证，验证和确认，审核和审核）; 和实际考虑（缺陷表征，软件质量测量和软件质量工具）。
  • Software Engineering Professional Practice（软件工程专业实践）：软件工程专业实践关注软件工程师必须具备的专业，负责和道德的软件工程知识，技能和态度。
  • Software Engineering Economics（软件工程经济学）：软件工程经济学知识域关注的是在业务环境中做出决策，以使技术决策与组织的业务目标保持一致。
  • Computing Foundations（计算基础）：计算基础知识域涵盖了提供软件工程实践所需的计算背景的基础主题。
  • Mathematical Foundations（数学基础）：数学基础知识域涵盖了提供软件工程实践所必需的数学背景的基础主题。
  • Engineering Foundations（工程基础）：工程基础知识域涵盖了提供软件工程实践所必需的工程背景的基础主题。

简单解释 CMMI 的五个级别。例如：Level 1 - Initial：无序，自发生产模式。

  1. 初始级：软件过程是无序的，有时甚至是混乱的，对过程几乎没有定义，成功取决于个人努力。管理是反应式的。

  2. 可管理级：建立了基本的项目管理过程来跟踪费用、进度和功能特性。制定了必要的过程纪律，能重复早先类似应用项目取得的成功经验。

  3. 已定义级：已将软件管理和工程两方面的过程文档化、标准化，并综合成该组织的标准软件过程。所有项目均使用经批准、剪裁的标准软件过程来开发和维护软件，软件产品的生产在整个软件过程是可见的。

  4. 量化管理级：分析对软件过程和产品质量的详细度量数据，对软件过程和产品都有定量的理解与控制。管理有一个作出结论的客观依据，管理能够在定量的范围内预测性能。

  5. 优化管理级：过程的量化反馈和先进的新思想、新技术促使过程持续不断改进。

用自己语言简述 SWEBOK 或 CMMI （约200字）

CMMI,能力成熟度模型，是一种改进软件开发的过程的方法，并以此来提升组织的绩效。通过这种模型，可以引导项目、部门或者一个大的部门的过程改进。因此对于软件工程这种极具系统性、复杂性的项目来说，CMMI可以极大的改进项目的过程，提高软件开发、运维过程的效率。同时，它也可以作为衡量一个企业软件开发水平的一个参考点。对于当前的CMMI，它主要应用的领域有三个方面，包括产品和服务的开发、服务创建管理和交付、产品和服务采购。CMMI模型有一个自己的框架，该框架涵盖了那些被一个组织的过程所覆盖的区域，如因果分析和解决，配置管理，集成的项目管理等。CMMI有五个级别，对于软件工程来说，不同的级别代表了一个组织软件工程的不同能力级别。CMMI级别越高，代表的其能力越大，代表一个开发企业的成熟度越高。
